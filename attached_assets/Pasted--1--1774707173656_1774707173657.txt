第二阶段：只修 6 个问题
====================

本轮只允许修以下 6 类问题，不允许扩展：

【1】启动配置强校验
要求：
- 启动时强校验以下环境变量：
  - TELEGRAM_BOT_TOKEN
  - TELEGRAM_ADMIN_CHAT_ID
  - 扫描周期变量（如果项目里有）
  - 其他启动必需变量
- 缺失时必须明确报错并退出
- 错误日志必须是中文
- 不要打印完整密钥
- 只允许打印：
  - 是否存在
  - 长度
  - 前 6 位 + 省略号

验收标准：
- 环境变量错误时，启动 3 秒内失败退出
- 日志能明确指出是哪一个配置缺失或格式不合法

【2】Telegram 启动链路稳定化
要求：
- 保留已经可用的 manual long polling 方案
- 不要再使用有兼容问题的 bot.launch() 作为主启动方式
- 确保服务重启时不会重复启动多个 polling 实例
- 增加优雅关闭逻辑
- 启动成功时输出中文日志：
  “Telegram 机器人已启动（手动长轮询）”

验收标准：
- 重启一次后只存在一个 polling 实例
- 日志中不再出现 AbortSignal 兼容报错

【3】风险引擎误伤修复
要求：
- 禁止使用简单 substring 方式匹配风险词
- 改为“规范化 + 单词边界匹配”或等价安全实现
- 必须修复 reward 命中 war 的误伤
- 保留风险词功能，但不能误杀常见正常词
- 移除以下链名的品牌误伤拦截（若当前仍在黑名单中）：
  - solana
  - ethereum
  - bitcoin
  - bnb

验收标准：
必须新增最小测试样例并输出结果：
- reward -> 不命中 war
- warfare -> 命中 war
- Solana meme token -> 不因 solana 被拦截
- BNB ecosystem -> 不因 bnb 被拦截

【4】ADMIN_CHAT_ID 自检能力
要求：
- 增加一个最小自检能力，二选一即可：
  A. 启动后向 ADMIN_CHAT_ID 发送一条测试消息（仅测试环境可开）
  B. 增加 /chatid 或 /whoami 命令，返回当前 chat id
- 日志要能明确区分：
  - token 无效
  - bot 已连接但无权限发群
  - chat id 错误
  - 群里还没加 bot
- 所有说明文字用中文

验收标准：
- 能明确判断 TELEGRAM_ADMIN_CHAT_ID 对不对
- 不再需要人工猜日志

【5】扫描任务去重与幂等
要求：
- 同一个候选代币在短时间内不能重复发送审批消息
- 增加最小幂等键，例如：
  tokenAddress + source + timeBucket
- 若项目里没有持久化，就先做内存级去重
- 如果已有数据库/缓存，就复用现有存储，不新加复杂基础设施

验收标准：
- 同一候选在一个扫描周期内最多发一次
- 连续两轮扫描不会因重复抓取而刷屏

【6】日志与可观测性补齐
要求：
- 扫描开始、扫描结束、候选数、拦截数、发送数都要有中文日志
- 每条被拦截的候选必须给出“被拦截原因”
- 每条成功发送审批消息必须打印：
  - token 标识
  - chat id
  - telegram message id
- 失败时必须打印失败原因
- 不要打印私钥、完整 token、完整敏感 headers

验收标准：
- 看日志就能知道：
  “抓到了多少 -> 被拦了多少 -> 发出了多少 -> 失败多少”
